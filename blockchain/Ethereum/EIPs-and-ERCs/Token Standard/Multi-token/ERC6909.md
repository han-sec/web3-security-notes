# ERC-6909: Minimal Multi-Token Interface

ERC-6909 is a minimal multi-token standard designed to manage multiple token types within a single contract. It is intentionally lightweight — stripping away the complexity of ERC-1155 (no batch operations, no safe-transfer hooks, no mandatory receiver callbacks) while retaining the two-tier approval model from ERC-20.

---

## Main Features

| Feature | Description |
|---|---|
| **Minimal surface area** | Only 7 functions. No callbacks, no batch ops, no metadata in core spec. |
| **Dual approval model** | Both operator-level (all tokens) and per-token-ID allowances — similar to ERC-20 `approve` + `permit`. |
| **Separate `transfer` / `transferFrom`** | Follows ERC-20 UX: `transfer` for self-initiated, `transferFrom` for delegated spending. |
| **No safe-transfer hooks** | No `onERC6909Received` check. Sends to contracts without reverting. |
| **ERC-165 introspection** | Contracts advertise support via `supportsInterface`. |
| **Extension via inheritance** | The spec defines optional extensions (Metadata, ContentURI, TokenSupply) that contracts can inherit and compose. |

> The "inherits context" refers to the extension model: ERC-6909 defines a **core** interface plus optional **extension interfaces** (e.g. `IERC6909Metadata`, `IERC6909TokenSupply`) that contracts inherit on top of the base. Each extension is its own interface with its own ERC-165 ID.

## Storage Layout

```solidity
// Token balances: id => owner => balance
mapping(uint256 id => mapping(address owner => uint256)) public balanceOf;

// Per-token-ID allowances: owner => spender => id => amount
mapping(address owner => mapping(address spender => mapping(uint256 id => uint256))) public allowance;

// Operator approvals (blanket): owner => operator => approved
mapping(address owner => mapping(address operator => bool)) public isOperator;
```

## Core Interface

```solidity
interface IERC6909 {
    // --- Events ---

    /// @notice Emitted on any token transfer (including mint/burn if modelled as from/to address(0))
    event Transfer(address caller, address indexed from, address indexed to, uint256 indexed id, uint256 amount);

    /// @notice Emitted when a per-token-ID allowance is set
    event Approval(address indexed owner, address indexed spender, uint256 indexed id, uint256 amount);

    /// @notice Emitted when an operator is granted or revoked blanket approval
    event OperatorSet(address indexed owner, address indexed operator, bool approved);

    // --- Views ---

    /// @notice Returns token balance of `owner` for token `id`
    function balanceOf(address owner, uint256 id) external view returns (uint256);

    /// @notice Returns the amount `spender` can transfer on behalf of `owner` for token `id`
    /// @dev   Does NOT reflect operator approvals — check isOperator separately
    function allowance(address owner, address spender, uint256 id) external view returns (uint256);

    /// @notice Returns whether `operator` is approved to transfer all tokens of `owner`
    function isOperator(address owner, address operator) external view returns (bool);

    // --- Mutations ---

    /// @notice Transfer `amount` of token `id` from caller to `to`
    /// @dev    Reverts if caller balance < amount
    function transfer(address to, uint256 id, uint256 amount) external returns (bool);

    /// @notice Transfer `amount` of token `id` from `from` to `to`
    /// @dev    Caller must be `from`, an approved operator of `from`,
    ///         or have sufficient per-token allowance from `from`.
    ///         If not an operator, decrements allowance by `amount`.
    function transferFrom(address from, address to, uint256 id, uint256 amount) external returns (bool);

    /// @notice Approve `spender` to transfer up to `amount` of token `id` on caller's behalf
    /// @dev    Overwrites (does not increment) the existing allowance
    function approve(address spender, uint256 id, uint256 amount) external returns (bool);

    /// @notice Grant or revoke blanket operator approval over all token IDs for caller
    function setOperator(address operator, bool approved) external returns (bool);
}
```

---

## Authorization Logic for `transferFrom`

When `transferFrom(from, to, id, amount)` is called:

1. If `msg.sender == from` → allowed (self-transfer).
2. Else if `isOperator[from][msg.sender] == true` → allowed, **no allowance consumed**.
3. Else if `allowance[from][msg.sender][id] >= amount` → allowed, **allowance decremented**.
4. Otherwise → revert.

> Unlike ERC-1155, there is no fallback to an ERC-20-style `approve` for all IDs at once unless `setOperator` is used. The per-token allowance and the operator flag are orthogonal.

## Optional Extensions (Inherited Interfaces)

ERC-6909 defines three optional extension interfaces. A contract signals support via ERC-165.

### 1. `IERC6909Metadata`
```solidity
interface IERC6909Metadata is IERC6909 {
    function name(uint256 id) external view returns (string memory);
    function symbol(uint256 id) external view returns (string memory);
    function decimals(uint256 id) external view returns (uint8);
}
```
- Per-token metadata. `decimals` is per-ID (unlike ERC-20 where it is contract-wide).
- Useful for multi-asset vaults where different IDs represent different assets with different precision.

### 2. `IERC6909ContentURI`
```solidity
interface IERC6909ContentURI is IERC6909 {
    function contractURI() external view returns (string memory);
    function tokenURI(uint256 id) external view returns (string memory);
}
```
- Off-chain metadata URIs, analogous to ERC-721's `tokenURI`.
- `contractURI` is contract-level; `tokenURI` is per-token-ID.

### 3. `IERC6909TokenSupply`
```solidity
interface IERC6909TokenSupply is IERC6909 {
    function totalSupply(uint256 id) external view returns (uint256);
}
```
- Per-ID total supply tracking. Not mandatory in core — implementations must opt in.

## ERC-6909 vs ERC-1155 vs ERC-20

| Property | ERC-20 | ERC-1155 | ERC-6909 |
|---|---|---|---|
| Multiple token types | No | Yes | Yes |
| `transfer` (self-initiated) | Yes | No (`safeTransferFrom` only) | Yes |
| `transferFrom` | Yes | Yes (via `safeTransferFrom`) | Yes |
| Per-token allowances | Yes | No | Yes |
| Operator approvals | No | Yes | Yes |
| Safe-transfer hooks | No | Yes (mandatory) | No |
| Batch transfers | No | Yes (mandatory in spec) | No |
| Receiver callback required | No | Yes (`onERC1155Received`) | No |
| Metadata in core spec | Yes (`name`, `symbol`, `decimals`) | No (optional URI) | No (optional extension) |
| Gas efficiency | High | Medium (callbacks) | High |

## Security Considerations

### Approval Race Condition
`approve` overwrites the existing allowance (same as ERC-20). The classic front-run race condition applies: set allowance to 0 first, then set the new value if changing a non-zero allowance.

### No Safe-Transfer Hooks
Tokens sent to a contract that does not handle them are not recovered. No `onERC6909Received` callback — this is by design for gas savings, but callers bear full responsibility.

### Operator vs Allowance Interaction
`isOperator` approval does **not** consume `allowance`. A spender who is both an operator and has an allowance will use the operator path (no decrement). This can cause confusion if the protocol assumes allowance is always decremented on spend.

### Per-ID `decimals` Risk
If the `Metadata` extension is implemented, different IDs may have different `decimals`. Protocols aggregating balances across IDs must normalize before arithmetic to avoid precision mismatches.

### Integer Overflow (pre-Solidity 0.8)
Supply and balance arithmetic should use checked math. In Solidity ≥0.8 this is default; in assembly-optimized implementations (e.g. Solmate) verify explicitly.
